# whodex Phase 1g — Facade + Notifications + TUI + serve + TOML config

> REQUIRED SUB-SKILL: superpowers:subagent-driven-development. **Testing law (AGENTS §10):** behavior/invariants via public interfaces; ALL real logic tested through the headless **facade** (not the UI); TUI tests are thin (Textual `Pilot` smoke + snapshots). Controller runs an INDEPENDENT full-gate checkpoint after every task.

**Goal:** the day-to-day surface. A headless `Whodex` **facade** exposes every read/write op; a `Notifier`/dispatch layer turns Changes+Reminders into deliveries (TUI sink now, Telegram/email later); a Textual **TUI** (priority queue, contact detail, contact-points, review/maintenance, log-interaction modal) calls only the facade; `whodex serve` runs one-shot+daemon on the same code; TOML config via `pydantic-settings`.

**Builds on:** 1a–1f. Reuses `run_sync`, engine queue/scoring, stores, edges, derived rows, ingestion app.

## Scope
In: `Whodex` facade (headless orchestration over engine/stores); `Notification` persistence + `Notifier` protocol + `NotificationDispatcher` + `TUINotifier`; Textual TUI (5 screens) via the facade; `whodex tui` + `whodex serve` (loop: run_sync→drain→dispatch, optional FastAPI mount + watchdog); TOML config (`pydantic-settings`, `whodex.toml`).
Out: Telegram/email sinks (Phase 2); web dashboard (Phase 3); rich graph viz.

## Invariants / behaviors (the tests — mostly via the FACADE, headless)
- **F1:** `facade.priority_queue()` returns the ranked people with why-now (same as `whodex queue`), excluding snoozed.
- **F2 log-interaction resets the clock:** `facade.log_interaction(person, ...)` appends an Interaction; the person's `overdue_ratio` drops → they leave the top of the queue (verified through `priority_queue` before/after).
- **F3 pin/snooze:** `facade.pin(p)` floors the score; `facade.snooze(p, until)` excludes them — observable via `priority_queue`.
- **F4 notifications:** a notable Change + a due Reminder produce `PENDING` notifications; `dispatch()` delivers via `TUINotifier` and marks delivered; dispatching again does NOT re-deliver (dedupe by `delivered_to`).
- **F5 review queue:** `facade.review_queue()` surfaces MergeCandidates + ConflictSuggestions + GraphRepairSuggestions; `apply_graph_repair(id)` records the UserAction (vault write is opt-in/existing).
- **F6 serve one-tick:** one `serve` tick = `run_sync + drain + dispatch` (same as a one-shot `sync`), idempotent.
- **F7 TUI smoke:** the priority-queue + contact-detail screens render (Textual `Pilot`) and a snapshot is stable; key bindings (`l` log, `s` snooze, `p` pin) invoke the facade. (Thin — logic lives in the facade.)
- **F8 config:** `pydantic-settings` loads `whodex.toml` (vault path, db path, cadence tiers, trust overrides, freshness TTLs, notifier toggles); env overrides; missing file → sensible defaults.

## Tasks

### Task 1: `Notification` persistence + Notifier/dispatch (F4)
Domain `Notification` (kind, entity_id, payload, dedupe_key, created_at, `delivered_to: set[str]`/list, state). Row+store (append-only + mark-delivered) mem+sqlite under contract. `Notifier` Protocol (`name`, `supports(n)`, `send(n) -> DeliveryResult`); `TUINotifier`/`ConsoleNotifier` (records/echoes). `NotificationDispatcher(notifiers)`: `dispatch(pending)` → for each PENDING not yet `delivered_to[sink]`, `send`, record delivery; idempotent. Behavioral F4 (no double-deliver). Gate checkpoint.

### Task 2: `Whodex` facade (F1/F2/F3/F5)
`app/facade.py` (new top-ish module; may import sync/engine/store/config) `Whodex` constructed from an `App`(+now/ids). Methods (implement what the TUI needs):
- reads: `priority_queue(limit, include_snoozed=False) -> list[RankedContact]`, `contact_detail(id) -> ContactDetail` (projected fields + freshness + contact points + timeline), `review_queue() -> list[ReviewItem]`, `people_at(id)`, `open_changes()`, `pending_notifications()`.
- writes (append event → reproject+persist via the shared `reproject_and_persist`): `log_interaction(id, kind, note, when=None)`, `pin(id, on)`, `snooze(id, until)`, `dismiss_reminder(fingerprint)`, `acknowledge_change(fingerprint)`, `apply_graph_repair(id)`, `set_cadence(id, days)`.
- orchestration: `sync()` (= run_sync), `dispatch_notifications()`.
DTOs: `RankedContact`, `ContactDetail`, `ReviewItem` (plain pydantic, never ORM rows). Behavioral facade tests F1/F2/F3/F5 (log-interaction drops a person from the top; pin floors; snooze excludes; review_queue aggregates). Gate checkpoint.

### Task 3: notifications from sync + `serve` one-tick (F4/F6)
Generate `Notification`s during `run_sync`/facade from notable un-acked Changes + due Reminders (dedupe_key = `kind:entity:fingerprint`); persist PENDING. `whodex serve` command: a loop calling `facade.sync()` + `facade.dispatch_notifications()` (one tick factored as a function `serve_tick(facade)` so it's testable without an infinite loop); `--once` for a single tick; optional FastAPI mount + watchdog are wired but the TESTABLE unit is `serve_tick`. Behavioral F6 (one tick == sync+dispatch, idempotent). Gate checkpoint.

### Task 4: TOML config via pydantic-settings (F8)
`config/settings.py`: `Settings(BaseSettings)` reading `whodex.toml` + env (`WHODEX_*`): `vault_path`, `db_path`, cadence tiers, trust overrides (merged onto DEFAULT_TRUST), freshness TTLs, notifier toggles. `build_app` accepts a `Settings` (or stays param-based with a `from_settings` adapter). Behavioral F8 (loads toml; env overrides; missing → defaults). Gate checkpoint.

### Task 5: Textual TUI — priority queue + contact detail (F7)
Add `textual` + `pytest-textual-snapshot` deps. `tui/app.py`: a `WhodexTUI(Textual App)` taking a `Whodex` facade. Screen 1 Priority Queue (ranked rows: name/tier/why-now/days; keys `enter`/`l`/`s`/`p`/`j`/`k`). Screen 2 Contact Detail (fields + freshness dots + contact points + timeline; `a` ack, `e` cadence). `whodex tui` CLI command builds the facade + runs the app. Tests: Textual `Pilot` smoke (app mounts, queue populated from a fake facade) + ONE snapshot; pressing `l`/`s`/`p` calls the facade (assert via a spy facade). Keep logic in the facade. Gate checkpoint.

### Task 6: TUI remaining screens + e2e + gate
Screens 3 Contact Points, 4 Review/Maintenance (accept/reject/apply), 5 Log-Interaction modal. `tests/test_e2e_phase1g.py`: end-to-end via the facade over a tmp vault+db — sync, queue ranked, log_interaction drops a person, pin/snooze observable, a notable change yields a pending notification dispatched once. Full gate + coverage. Independent gate verify before merge.
